# /ship — Финализация фичи (release workflow)

Этот скилл автоматизирует стандартный 8-шаговый цикл выпуска фичи из ARCHITECTURE.md.

## Когда использовать

После завершения реализации фичи и проверки в браузере — перед мержем в master.

## Шаги выполнения

### 1. Определить тип версии

Прочитай текущую версию из `src/data/changelog.js` (поле `GAME_VERSION` — это единственный источник истины для версии игры).

Спроси пользователя (если не передана как аргумент `$ARGUMENTS`):
- `patch` — баг-фикс, баланс-правки, мелкие улучшения
- `minor` — новая механика или заметный контент (новый стат, новая система)
- `major` — рестарт прогрессии, кардинальная смена геймплея

### 2. Bump версии в трёх местах

Вычисли новую версию (semver) и обнови:

1. **`src/data/changelog.js`** — поле `export const GAME_VERSION = 'X.Y.Z'`
2. **`package.json`** — поле `"version": "X.Y.Z"`

### 3. Обновить `src/data/changelog.js` — внутриигровой changelog

Это **основной** файл "Что нового": SettingsMenu читает его и показывает автопопап при смене версии.

Добавь новый объект в начало массива `CHANGELOG`:

```js
{
  version: 'X.Y.Z',
  date: 'YYYY-MM-DD',
  entries: [
    { type: 'new',     text: '🆕 Описание новой фичи.' },
    { type: 'changed', text: '🔄 Описание изменения.' },
    { type: 'fixed',   text: '🔧 Описание исправления.' },
    { type: 'balance', text: '⚖️ Описание балансного изменения.' },
  ],
},
```

Допустимые типы: `'new' | 'changed' | 'fixed' | 'balance'`. Используй только нужные — пустых entries не добавляй.

**Важно:** текст entries — это то, что видит игрок в игре. Пиши по-русски, с эмодзи, кратко и понятно для игрока (не технически).

### 4. Обновить `CHANGELOG.md` — devlog для GitHub

Прочитай `CHANGELOG.md`. Вставь новый блок **сразу после строки `---` в начале** (перед последней версией):

```markdown
## [X.Y.Z] — YYYY-MM-DD

### Добавлено
- ...

### Изменено
- ...

### Исправлено
- ...
```

Используй сегодняшнюю дату. Пустые секции не добавляй. Здесь можно быть более техническим.

### 5. Обновить ARCHITECTURE.md

Прочитай `ARCHITECTURE.md`:
- Найди задачи в бэклоге (`- [ ]`), которые реализованы в этой фиче, и отметь `- [x]`
- Если менялась файловая структура или размеры файлов — обнови таблицу в секции «Файловая структура»
- Если добавлена новая механика — добавь описание в соответствующую секцию

### 6. Git-коммит и мерж

```bash
git status  # посмотреть что изменилось
git add src/data/changelog.js package.json CHANGELOG.md ARCHITECTURE.md
git add -u  # остальные рабочие файлы
```

Предложи сообщение коммита:
```
release: vX.Y.Z — <краткое описание фичи>
```

После подтверждения:
```bash
git commit -m "release: vX.Y.Z — <описание>"
git checkout master
git merge feature/<name>
git push
```

Спроси имя ветки если неизвестно (`git branch --show-current`).

## Важно

- **Два changelog-файла** — `src/data/changelog.js` (игровой UI) и `CHANGELOG.md` (GitHub). Оба обязательны
- `GAME_VERSION` в `src/data/changelog.js` определяет автопопап "Что нового" — если не обновить, игрок не увидит изменения
- Не форс-пушь и не используй `--no-verify`
- Если есть незакоммиченные изменения в рабочих файлах — включи их в коммит
